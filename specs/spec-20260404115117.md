# SubLink MVP Spec — 2026-04-04 11:51

Privacy-preserving subscription protocol built on Unlink SDK. No smart contracts. Backend + Unlink transfers + mock creator sites.

---

## Roles

- **Creator**: registers on SubLink, creates subscription plans, tracks earnings
- **Subscriber**: discovers plans on 3rd-party creator sites, subscribes by funding a dedicated Unlink account, manages subscriptions on SubLink UI (future)
- **3rd-party Creator Site**: gates content behind SubLink subscriptions, verifies access via SubLink API

---

## Core Flow

### 1. Creator Setup
1. Creator derives an Unlink account from their EVM key
2. Registers with SubLink backend: `POST /creators` (evmAddress, unlinkAddress, name)
3. Gets back `creatorId` + `apiKey` (used by their site to verify subscribers)
4. Creates a plan: `POST /plans` (amount, interval, name)

### 2. Subscriber Subscribes (on 3rd-party site)
1. Subscriber hits a gated endpoint on creator's site
2. Gets HTTP 402 with discovery body (planId, amount, interval, SubLink API URL)
3. Subscriber derives a **dedicated Unlink account** for this subscription:
   - `derivationIndex = Number(BigInt(keccak256(planId)) % 1_000_000n)`
   - `seed = keccak256(walletClient.signMessage("sublink:{planId}"))`
   - `account = unlinkAccount.fromSeed({ seed: hexToBytes(seed) })`
4. Registers the dedicated account with Unlink (`ensureRegistered()`)
5. Deposits USDC into the dedicated account (`deposit()`)
6. Calls `POST /subscribe` with planId, subscriberEvmAddress, dedicated account's unlinkAddress, and serialized AccountKeys
7. Backend executes **immediate first charge** (transfer from dedicated account to creator's Unlink address)
8. Returns subscriptionId + first charge result

### 3. Recurring Charges (Cron)
1. Backend cron runs every 60s (configurable, shorter for demo)
2. Selects subscriptions where `next_charge_at <= now AND status = 'active' AND consecutive_failures < 3`
3. For each:
   - Reconstruct Unlink client from stored AccountKeys
   - Pre-check balance via `getBalances()`
   - If balance < amount: **soft-fail** (increment consecutive_failures, push next_charge_at)
   - If balance ok: `transfer()` to creator's Unlink address
   - Poll `pollTransactionStatus()` until terminal
   - On success: reset failures, update charge_count, total_spent, next_charge_at
   - On 3 consecutive failures: set status = `cancelled_by_failure`
   - If spending_cap > 0 and total_spent >= cap: set status = `completed`

### 4. Access Verification (3rd-party site)
1. Subscriber creates a Bearer token: `{subscriptionId}.{expiry}.{signature}`
   - message = `{subscriptionId}:{expiry}`, signed with subscriber's EVM key
2. Subscriber sends request to creator's site with `Authorization: Bearer {token}`
3. Creator's site calls SubLink `GET /verify/:planId` with the token + `X-Api-Key`
4. Backend: parse token, check expiry (30s skew), recover signer, lookup subscription, verify active + plan matches + signer matches
5. Returns `{ valid: true/false }`

### 5. Cancellation
- **Explicit**: subscriber calls `DELETE /subscriptions/:id`
- **Implicit (drain)**: subscriber withdraws all USDC from dedicated account. Next charge soft-fails. After 3 failures, auto-cancelled.

## API Routes

### Platform API

| Method | Path | Description |
|--------|------|-------------|
| GET | /health | `{ ok: true }` |
| POST | /creators | Register creator. Body: `{ evmAddress, unlinkAddress, name }`. Returns `{ id, apiKey }` |
| GET | /creators/:id | Get creator (no apiKey in response) |
| POST | /plans | Create plan. Body: `{ creatorId, name, amount, intervalSeconds, description?, spendingCap? }`. Returns plan |
| GET | /plans | List active plans. Query: `?creatorId=` optional |
| GET | /plans/:id | Plan + creator info (unlinkAddress needed for subscribe flow) |
| POST | /subscribe | Subscribe. Body: `{ planId, subscriberEvmAddress, unlinkAddress, accountKeysJson }`. Executes first charge. Returns `{ subscriptionId, firstCharge: { txId, status } }` |
| GET | /subscriptions | List subscriptions. Query: `?subscriberEvmAddress=` or `?planId=` or `?creatorId=` |
| DELETE | /subscriptions/:id | Cancel subscription |
| GET | /charges/:subscriptionId | Charge history |
| POST | /admin/run-cron | Manually trigger cron executor (for testing) |

### Verification API (called by 3rd-party sites)

| Method | Path | Headers | Description |
|--------|------|---------|-------------|
| GET | /verify/:planId | `Authorization: Bearer {token}`, `X-Api-Key: {creatorApiKey}` | Verify subscriber access |

**200 response:**
```json
{ "valid": true, "subscriptionId": "...", "subscriberEvmAddress": "0x...", "planId": "..." }
```

**402 response (no/invalid token):**
```json
{
  "valid": false,
  "error": "Subscription required",
  "sublink": {
    "planId": "...",
    "plan": { "name": "Pro", "amount": "500000", "intervalSeconds": 2592000 },
    "sublinkApiUrl": "http://localhost:3000"
  }
}
```

## AccountKeys Serialization

`AccountKeys` contains `bigint` and `Uint8Array` which aren't JSON-safe.

```typescript
// Serialized format (all JSON-safe strings)
interface SerializedAccountKeys {
  spendingPrivateKey: string;          // bigint as decimal string
  spendingPublicKey: [string, string]; // bigints as decimal strings
  viewingPrivateKey: string;           // Uint8Array as hex
  viewingPublicKey: string;            // Uint8Array as hex
  nullifyingKey: string;               // bigint as decimal
  masterPublicKey: string;             // bigint as decimal
  address: string;                     // unlink1... passthrough
}
```

Functions: `serializeAccountKeys(keys) → string`, `deserializeAccountKeys(json) → AccountKeys`

## Cron Executor Details

- Runs inside backend process via `setInterval`
- Default interval: 60s (configurable via `CRON_INTERVAL_MS` env)
- Mutex flag prevents overlapping runs
- For each due subscription:
  1. Deserialize AccountKeys
  2. `createUnlink({ account: unlinkAccount.fromKeys(keys), engineUrl, apiKey })` — no `evm` needed for transfers
  3. `getBalances({ token: USDC })` — pre-check
  4. `transfer({ recipientAddress: creator.unlinkAddress, token: USDC, amount: plan.amount })`
  5. `pollTransactionStatus(txId, { intervalMs: 3000, timeoutMs: 180000 })`
  6. Update DB based on result
- Sequential processing (one subscription at a time) to avoid API rate limits

## Privacy Properties

| What | Visible to | Hidden from |
|------|-----------|-------------|
| Creator's plans | Everyone | (public by design) |
| Who subscribes to whom | SubLink backend only | Chain observers, other users |
| Payment amounts | SubLink backend only | Chain observers |
| Payment timing | SubLink backend only | Chain observers |
| Creator's total earnings | SubLink backend only | Chain observers |
| Subscriber's total spending | SubLink backend only | Chain observers |

All payments are Unlink private transfers (sender, recipient, amount, token hidden on-chain via ZK proofs). The backend is the single trust point — TEE upgrade path for later.
