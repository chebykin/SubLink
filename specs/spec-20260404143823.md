# SubLink MVP Spec — 2026-04-04 14:38 (rev 3)

Privacy-preserving subscription protocol built on Unlink SDK. No smart contracts. Backend + Unlink transfers + mock creator sites.

Changes from rev 2:
- subscriber EVM addresses are no longer stored in the backend database
- subscribers use a deterministic auth key derived from their wallet
- Bearer tokens are signed with the derived auth key, not the wallet EVM key
- verification responses no longer expose `subscriberEvmAddress`

This spec is the next target design. Backend implementation is pending.

---

## Roles

- **Creator**: registers on SubLink, creates subscription plans, tracks earnings
- **Subscriber**: discovers plans on 3rd-party creator sites, subscribes by funding a dedicated Unlink account, authenticates with a derived auth key
- **3rd-party Creator Site**: gates content behind SubLink subscriptions, verifies access via SubLink API

---

## Core Flow

### 1. Creator Setup
1. Creator derives an Unlink account from their EVM key
2. Registers with SubLink backend: `POST /creators` (`evmAddress`, `unlinkAddress`, `name`)
3. Gets back `creatorId` + `apiKey` (used by their site to verify subscribers)
4. Creates a plan: `POST /plans` (`amount`, `interval`, `name`)

### 2. Subscriber Subscribes (on 3rd-party site)
1. Subscriber hits a gated endpoint on creator's site
2. Gets HTTP 402 with discovery body (`planId`, `amount`, `interval`, `SubLink API URL`)
3. Subscriber derives a **dedicated Unlink account** for this subscription:
   - `derivationIndex = Number(BigInt(keccak256(toHex(planId))) % 1_000_000n)`
   - `seed = keccak256(walletClient.signMessage("sublink:{planId}"))`
   - `account = unlinkAccount.fromSeed({ seed: hexToBytes(seed), accountIndex: derivationIndex })`
4. Subscriber derives a **deterministic auth key** for all of their subscriptions:
   - `authDerivationMessage = "sublink-auth-v1"`
   - `authSignature = walletClient.signMessage(authDerivationMessage)`
   - `authSeed = keccak256(authSignature)`
   - `authPrivateKey = authSeed`
   - `authAccount = privateKeyToAccount(authPrivateKey)`
   - `authPublicKey = secp256k1.getPublicKey(authPrivateKey, false)` as uncompressed hex (`0x04...`)
   - `authKeyId = lower(authAccount.address)`
5. Subscriber creates a subscribe proof signed with the auth key:
   - `authProofMessage = "sublink-subscribe-v1:{planId}:{unlinkAddress}:{authKeyId}"`
   - `authProof = authAccount.signMessage(authProofMessage)`
6. Registers the dedicated account with Unlink (`ensureRegistered()`)
7. Deposits USDC into the dedicated account (`deposit()`)
8. Calls `POST /subscribe` with `planId`, dedicated account `unlinkAddress`, serialized `AccountKeys`, `authPublicKey`, `authKeyId`, and `authProof`
9. Backend verifies the auth proof, verifies `authKeyId` matches `authPublicKey`, enforces uniqueness on `(planId, authKeyId)`, then executes the **immediate first charge**
10. Returns `subscriptionId` + first charge result

### 3. Recurring Charges (Cron)
1. Backend cron runs every 60s (configurable, shorter for demo)
2. Selects subscriptions where `next_charge_at <= now AND status = 'active' AND consecutive_failures < 3`
3. For each:
   - Reconstruct Unlink client from stored AccountKeys
   - Pre-check balance via `getBalances()`
   - If balance < amount: **soft-fail** (increment `consecutive_failures`, push `next_charge_at`)
   - If balance ok: `transfer()` to creator's Unlink address
   - Poll `pollTransactionStatus()` until terminal
   - On success: reset failures, update `charge_count`, `total_spent`, `next_charge_at`
   - On 3 consecutive failures: set status = `cancelled_by_failure`
   - If `spending_cap > 0` and `total_spent >= cap`: set status = `completed`

### 4. Access Verification (3rd-party site)
1. Subscriber creates a Bearer token: `{subscriptionId}.{expiry}.{signature}`
   - `bearerMessage = "sublink-bearer-v1:{subscriptionId}:{expiry}"`
   - `expiry` is a UTC unix timestamp in seconds
   - `signature = authAccount.signMessage(bearerMessage)`
2. Subscriber sends request to creator's site with `Authorization: Bearer {token}`
3. Creator's site calls SubLink `GET /verify/:planId` with the token + `X-Api-Key`
4. Backend parses the token, checks expiry (30s skew), recovers the auth signer, loads the subscription, verifies:
   - subscription exists
   - subscription is active
   - subscription `planId` matches
   - recovered auth signer matches `subscription.authKeyId`
5. Returns `{ valid: true/false }`

### 5. Subscriber API Authentication
- `GET /subscriptions` accepts any valid auth-key Bearer token and returns all subscriptions for the caller's `authKeyId`
- `DELETE /subscriptions/:id` accepts any valid auth-key Bearer token and returns 403 unless the caller's `authKeyId` owns the subscription
- `GET /charges/:subscriptionId` accepts any valid auth-key Bearer token and returns 403 unless the caller's `authKeyId` owns the subscription

### 6. Cancellation
- **Explicit**: subscriber calls `DELETE /subscriptions/:id`
- **Implicit (drain)**: subscriber withdraws all USDC from the dedicated account. Next charge soft-fails. After 3 failures, auto-cancelled.

## Deterministic Auth Key

### Design
- One auth key per subscriber wallet in this rev
- Auth key is derived, not randomly generated and stored
- Backend stores the auth key identity, never the subscriber's wallet EVM address
- Subscriber can re-derive the auth key by signing the same message again with the same wallet

### Canonical Identifier
- `authKeyId = lower(authAccount.address)`
- `authKeyId` is the main lookup key used by subscriber endpoints and duplicate prevention

### Exact Encoding Rules
- All signed messages are UTF-8 strings with no trailing newline
- `signMessage(...)` refers to the wallet's normal EIP-191 / personal-sign flow
- `keccak256(authSignature)` means hashing the raw 65-byte signature bytes returned by `signMessage`
- `authPrivateKey` must be interpreted as a secp256k1 private key in hex form
- MVP assumption: supported wallets are EOAs that return a stable signature for the same derivation message
- Smart contract wallets are out of scope for this rev

### Stored Material
- `authPublicKey` is stored for explicit auditability and subscribe-proof verification
- `authKeyId` is stored for indexing, uniqueness, and quick comparisons

### Security / UX Tradeoff
- This rev intentionally keeps one auth key per wallet, so the backend can still link that subscriber's subscriptions to each other
- The benefit is simple UX: one deterministic identity, no extra key storage
- If less cross-creator linkability is needed later, a future rev can scope the auth domain string per creator:
  - example: `sublink-auth-v1:{creatorId}`

### Wallet Compatibility Assumption
- This design assumes supported wallets produce stable signatures for the same `signMessage("sublink-auth-v1")` input
- That is expected for standard EOAs in the planned demo flow
- If wallet variance shows up in practice, a future rev must switch to explicit auth-key storage or a different derivation mechanism

## Duplicate Prevention

- This rev does **not** use a separate `blinded_id`
- Duplicate prevention is handled by `UNIQUE(plan_id, auth_key_id)`
- Rationale: once a single deterministic auth key represents the subscriber, a plan-scoped blinded identifier is redundant for MVP

## Authentication

### Bearer Token (subscriber endpoints)
Endpoints that access subscriber data require `Authorization: Bearer {token}`. The backend recovers the caller's **auth key identity** from the token and uses that to scope access:

- `GET /subscriptions` — returns only subscriptions for the recovered `authKeyId`
- `DELETE /subscriptions/:id` — 403 if recovered `authKeyId` does not own the subscription
- `GET /charges/:subscriptionId` — 403 if recovered `authKeyId` does not own the subscription
- `GET /verify/:planId` — verifies `subscriptionId`, expiry, active status, plan match, and `authKeyId` match
- `GET /subscriptions` and `DELETE /subscriptions/:id` responses must never include `accountKeysEncrypted`

Returns 401 if token is missing, malformed, or expired.

### Subscribe Proof
`POST /subscribe` also requires a one-time auth-key proof inside the body:

- `authProofMessage = "sublink-subscribe-v1:{planId}:{unlinkAddress}:{authKeyId}"`
- `authProof = signMessage(authProofMessage)` using the derived auth key

Backend must verify:
- `authProof` is valid
- recovered signer matches `authKeyId`
- `authKeyId` is consistent with the supplied `authPublicKey`

### Bearer Token Format
- token format = `{subscriptionId}.{expiry}.{signature}`
- `bearerMessage = "sublink-bearer-v1:{subscriptionId}:{expiry}"`
- `expiry` must be an integer UTC unix timestamp in seconds
- `signature` is generated by the derived auth key
- backend expiry policy remains:
  - max token age = `BEARER_TOKEN_MAX_AGE_SECONDS`
  - allowed clock skew = `BEARER_TOKEN_CLOCK_SKEW_SECONDS`

### API Key (creator verification)
- `GET /verify/:planId` — requires `X-Api-Key` header matching the plan's creator. Returns 401 if invalid.

### Creator Verification Semantics
- A creator site can verify whether a subscriber currently has valid access at any time by calling `GET /verify/:planId` with:
  - the creator's `X-Api-Key`
  - a fresh subscriber Bearer token
- `/verify` answers **current validity**, not historical payment history
- In this rev, there is no creator-only endpoint to enumerate subscribers, inspect all charges for a plan, or ask "has this unseen wallet ever paid?" without a subscriber-provided token

### Admin Secret (admin endpoints)
- `POST /admin/run-cron` — requires `x-admin-key` header matching `ADMIN_SECRET` env var. Returns 403 if the secret is missing or invalid.

### Public (no auth)
- `GET /health`
- `POST /creators`
- `POST /plans`
- `GET /plans`, `GET /plans/:id`
- `GET /creators/:id`

## API Routes

### Platform API

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| GET | /health | none | `{ ok: true }` |
| POST | /creators | none | Register creator. Body: `{ evmAddress, unlinkAddress, name }`. Returns `{ id, apiKey }` |
| GET | /creators/:id | none | Get creator (no `apiKey` in response) |
| POST | /plans | none | Create plan. Body: `{ creatorId, name, amount, intervalSeconds, description?, spendingCap? }`. Returns plan |
| GET | /plans | none | List active plans. Query: `?creatorId=` optional. Returns `{ plans: [...] }` |
| GET | /plans/:id | none | Plan + creator info. Returns plan with creator object |
| POST | /subscribe | subscribe proof | Subscribe. Body: `{ planId, unlinkAddress, accountKeysJson, authKeyId, authPublicKey, authProof }`, where `authProof` signs `"sublink-subscribe-v1:{planId}:{unlinkAddress}:{authKeyId}"`. Executes first charge. Returns `{ subscriptionId, firstCharge: { txId, status } }` |
| GET | /subscriptions | Bearer | List caller's subscriptions. Returns `{ subscriptions: [...] }` with `accountKeysEncrypted` stripped from every item |
| DELETE | /subscriptions/:id | Bearer | Cancel subscription. 403 if not owner |
| GET | /charges/:subscriptionId | Bearer | Charge history. 403 if not owner. Returns `{ charges: [...] }` |
| POST | /admin/run-cron | x-admin-key | Manually trigger cron executor. Returns `CronRunSummary` |

### Verification API (called by 3rd-party sites)

| Method | Path | Headers | Description |
|--------|------|---------|-------------|
| GET | /verify/:planId | `Authorization: Bearer {token}`, `X-Api-Key: {creatorApiKey}` | Verify current subscriber access using auth-key Bearer token |

**200 response:**
```json
{ "valid": true, "subscriptionId": "...", "planId": "..." }
```

**402 response (no/invalid token):**
```json
{
  "valid": false,
  "error": "Subscription required",
  "sublink": {
    "planId": "...",
    "plan": { "name": "Pro", "amount": "500000", "intervalSeconds": 2592000 },
    "sublinkApiUrl": "http://localhost:3000"
  }
}
```

## AccountKeys Serialization

`AccountKeys` contains `bigint` and `Uint8Array` which are not JSON-safe.

```typescript
interface SerializedAccountKeys {
  spendingPrivateKey: string;
  spendingPublicKey: [string, string];
  viewingPrivateKey: string;
  viewingPublicKey: string;
  nullifyingKey: string;
  masterPublicKey: string;
  address: string;
}
```

Functions: `serializeAccountKeys(keys) -> string`, `deserializeAccountKeys(json) -> AccountKeys`

## Cron Executor Details

- Runs inside backend process via `setInterval`
- Default interval: 60s (configurable via `CRON_INTERVAL_MS` env)
- Mutex flag prevents overlapping runs
- Per-account transfer lock prevents concurrent transfers to the same dedicated account
- For each due subscription:
  1. Deserialize AccountKeys
  2. `createUnlink({ account: unlinkAccount.fromKeys(keys), engineUrl, apiKey })`
  3. `getBalances({ token: USDC })`
  4. `transfer({ recipientAddress: creator.unlinkAddress, token: USDC, amount: plan.amount })`
  5. `pollTransactionStatus(txId, { intervalMs: 3000, timeoutMs: 180000 })`
  6. Update DB based on result
- Sequential processing (one subscription at a time) to avoid API rate limits

## Migration / Compatibility Notes

- Existing rev 2 subscriptions keyed by `subscriber_evm_address` are not directly compatible with this rev
- Existing EVM-signed Bearer tokens become invalid under this rev
- Existing `/verify` success payloads that include `subscriberEvmAddress` are removed
- Backend migration requires:
  - schema change from `subscriber_evm_address` to `auth_key_id` + `auth_public_key`
  - new subscribe body fields: `authKeyId`, `authPublicKey`, `authProof`
  - subscriber endpoints to scope by `authKeyId`
  - Bearer token verification to recover the auth key signer
- Because the backend never stored auth keys for old subscriptions, rev 2 subscriptions cannot be losslessly backfilled without subscriber participation
- MVP migration path is either:
  - destructive reset for demo data, or
  - require subscribers to re-subscribe under the new auth scheme

---

## Summary

This rev removes the need to persist subscriber wallet EVM addresses while keeping subscriber UX simple:

- dedicated Unlink account remains per plan
- auth identity becomes deterministic per wallet
- backend stores `authKeyId` / `authPublicKey`, not the real wallet address
- Bearer tokens and subscriber endpoints use the auth key
- duplicate prevention remains simple via `UNIQUE(plan_id, auth_key_id)`
