Privacy-preserving subscription protocol built on Unlink SDK. No smart contracts. Backend + Unlink private transfers + creator-controlled verification.

## 1. Goal

SubLink lets creators gate access to recurring paid content without exposing subscriber wallet addresses or public payment trails.

Key properties:

- No smart contracts for billing logic
- Unlink private transfers on Base Sepolia
- Creator sites verify access through SubLink backend
- Subscriber identity at verification time is a derived auth key, not the funding wallet
- Access is entitlement-based, not status-string-based

## 2. Actors

- Creator: registers with SubLink, creates plans, receives recurring payments
- Subscriber: funds a dedicated Unlink account per plan and authenticates with a derived auth key
- Creator Site: third-party website that protects content and calls SubLink `/verify/:planId`
- SubLink Backend: stores plans/subscriptions, executes charges, verifies entitlement

## 3. Chain / Tokens

- Chain: Base Sepolia
- RPC: `https://sepolia.base.org`
- Explorer: `https://sepolia-explorer.base.org`
- Token: USDC `0x036CbD53842c5426634e7929541eC2318f3dCF7e`

## 4. Identity Model

### 4.1 Dedicated Unlink account

Each subscription uses its own dedicated Unlink account.

- `derivationIndex = Number(BigInt(keccak256(toHex(planId))) % 1_000_000n)`
- `seed = keccak256(walletClient.signMessage("sublink:{planId}"))`
- `unlinkAccount = unlinkAccount.fromSeed({ seed, accountIndex: derivationIndex })`

This isolates funds per plan.

### 4.2 Auth key

Subscribers derive a deterministic auth key from their EVM wallet:

- sign message: `sublink-auth-v1`
- hash the signature with `keccak256`
- turn the hash into a secp256k1 private key

`authKeyId` is the lowercase EVM address of this auth key.

The auth key is used only for proofs and bearer tokens, never for moving Unlink funds.

## 5. Subscription Access Model

SubLink distinguishes lifecycle state from access entitlement.

### 5.1 Entitlement source of truth

The source of truth for access is:

- `paidThroughAt`

A subscriber has valid access if and only if:

- bearer token is valid
- subscription belongs to the requested `planId`
- recovered signer matches `subscription.authKeyId`
- `paidThroughAt` exists
- `paidThroughAt > now`

### 5.2 Subscription statuses

- `pending_activation`
  - initial charge has not succeeded yet
  - no entitlement
  - eligible for billing retries
- `active`
  - renewable subscription with current or future billing entitlement
- `past_due`
  - a renewal charge failed and entitlement was not extended
  - eligible for billing retries
- `cancelled`
  - subscriber requested stop
  - no further renewals
  - existing access remains valid until `paidThroughAt`
- `cancelled_by_failure`
  - stopped after maximum consecutive failures
  - no further renewals
- `completed`
  - spending cap reached
  - no further renewals
  - existing access remains valid until `paidThroughAt`

### 5.3 Chargeable statuses

Billing cron may charge subscriptions in:

- `pending_activation`
- `active`
- `past_due`

## 6. Core Flows

### 6.1 Creator setup

1. Creator derives or supplies one Unlink account
2. Registers with SubLink: `POST /creators`
3. Receives `creatorId` and `apiKey`
4. Creates a plan with `POST /plans`

### 6.2 Initial subscription

1. Subscriber visits creator site
2. Site prompts for subscription if verification fails
3. Subscriber derives dedicated Unlink account for `planId`
4. Subscriber derives auth key
5. Subscriber funds dedicated account with USDC
6. Subscriber signs subscribe proof:
   - `sublink-subscribe-v1:{planId}:{unlinkAddress}:{authKeyId}`
7. Subscriber calls `POST /subscribe` with:
   - `planId`
   - `unlinkAddress`
   - `accountKeysJson`
   - `authKeyId`
   - `authPublicKey`
   - `authProof`
8. Backend verifies proof and creates a `pending_activation` subscription
9. Backend immediately attempts the first charge
10. On success:
    - status becomes `active`
    - `paidThroughAt = chargedAt + interval`
    - `nextChargeAt = paidThroughAt`
    - returns `201`
11. On failure:
    - status remains `pending_activation`
    - `paidThroughAt = null`
    - returns `402`

### 6.3 Renewal billing

Cron selects subscriptions where:

- `status IN ('pending_activation', 'active', 'past_due')`
- `next_charge_at <= now`
- `consecutive_failures < MAX_CONSECUTIVE_FAILURES`

For each attempt:

- Create pending charge row
- Execute private transfer from dedicated Unlink account to creator Unlink address

On successful charge:

- `totalSpent += plan.amount`
- `chargeCount += 1`
- `consecutiveFailures = 0`
- `lastChargedAt = chargedAt`
- `paidThroughAt = max(previousPaidThroughAt, chargedAt) + interval`
- `nextChargeAt = paidThroughAt`
- status becomes:
  - `completed` if spending cap reached
  - otherwise `active`

On failed charge:

- `totalSpent` unchanged
- `chargeCount` unchanged
- `paidThroughAt` unchanged
- `consecutiveFailures += 1`
- `nextChargeAt = chargedAt + interval`
- status becomes:
  - `pending_activation` if no successful charge has ever created entitlement
  - `past_due` if entitlement existed but was not extended
  - `cancelled_by_failure` on the max failure threshold

### 6.4 Verification

1. Subscriber creates bearer token:
   - format: `{subscriptionId}.{expiry}.{signature}`
   - message: `sublink-bearer-v1:{subscriptionId}:{expiry}`
2. Subscriber sends request to creator site with `Authorization: Bearer {token}`
3. Creator site calls `GET /verify/:planId` with:
   - `Authorization: Bearer {token}`
   - `X-Api-Key: {creatorApiKey}`
4. Backend verifies:
   - plan exists
   - creator API key matches the plan owner
   - token format and expiry are valid
   - recovered signer matches the subscription `authKeyId`
   - subscription `planId` matches
   - `paidThroughAt > now`
5. If yes: return `{ valid: true, subscriptionId, planId }`
6. If not: return `402` discovery payload

### 6.5 Cancellation

Subscriber can cancel chargeable subscriptions:

- `active`
- `past_due`
- `pending_activation`

Cancellation:

- sets status to `cancelled`
- stops future renewals
- does not remove existing entitlement

## 7. API

### Public / creator setup

| Method | Path | Auth | Notes |
| --- | --- | --- | --- |
| POST | `/creators` | none | Register creator. Body: `{ evmAddress, unlinkAddress, name }`. Returns `{ id, apiKey }` |
| GET | `/creators/:id` | none | Returns creator public info only |
| GET | `/creators/by-evm/:evmAddress` | none | Returns creator public info only |
| POST | `/plans` | none | Create plan. Body: `{ creatorId, name, amount, intervalSeconds, description?, spendingCap? }` |
| GET | `/plans` | none | List active plans |
| GET | `/plans/:id` | none | Plan with creator info |

### Subscriber

| Method | Path | Auth | Notes |
| --- | --- | --- | --- |
| POST | `/subscribe` | subscribe proof | Creates `pending_activation` subscription, attempts immediate first charge. Returns `201` on success or `402` on failed initial charge |
| GET | `/subscriptions` | Bearer | Returns only subscriptions owned by recovered `authKeyId` |
| DELETE | `/subscriptions/:id` | Bearer | Cancels subscription if caller owns it and subscription is chargeable |
| GET | `/charges/:subscriptionId` | Bearer | Returns charge history only if caller owns the subscription |

### Creator verification

| Method | Path | Auth | Notes |
| --- | --- | --- | --- |
| GET | `/verify/:planId` | `Authorization: Bearer {token}` + `X-Api-Key` | Returns `200` only when entitlement is currently valid |

### Admin

| Method | Path | Auth | Notes |
| --- | --- | --- | --- |
| POST | `/admin/run-cron` | `X-Admin-Key` | Run one cron cycle manually |

## 9. Security / Privacy Rules

- Creator sites never receive subscriber wallet addresses
- Creator sites never receive Unlink account keys
- Backend stores dedicated Unlink account keys so it can execute recurring charges
- Auth keys are separate from Unlink spending keys
- Bearer tokens are stateless and recover `authKeyId` from signature
- Creator verification requires both creator ownership proof (`X-Api-Key`) and subscriber proof (`Bearer`)

## 10. Verification Guarantees

Creator verification answers:

- "Does this caller currently have paid entitlement for this plan?"

It does not answer:

- who the subscriber is in public wallet terms
- how much balance remains in the dedicated Unlink account
- whether the caller is set to auto-renew forever

## 11. Non-Goals for This Revision

- Smart contract billing
- Public on-chain subscription registries
- Creator-facing enumeration of all subscribers through the verification endpoint
- Lossless migration from old wallet-address-based subscription rows without subscriber participation
