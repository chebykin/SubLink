# SubLink Spec — 2026-04-04 20:43

Privacy-preserving subscription protocol built on Unlink SDK. No smart contracts. Backend + Unlink private transfers + creator-controlled verification.

## 1. Goal

SubLink lets creators gate recurring paid access without exposing subscriber wallet addresses or public payment trails.

Core properties:

- Unlink private transfers on Base Sepolia
- Creator sites verify access through SubLink backend
- Subscriber authentication uses a derived auth key, not the funding wallet
- Access is entitlement-based via `paidThroughAt`
- Initial activation failures are retryable without creating duplicate subscriptions

## 2. Actors

- Creator: registers, creates plans, receives recurring private transfers
- Subscriber: funds a dedicated Unlink account per plan and authenticates with a derived auth key
- Creator Site: protects content and calls `GET /verify/:planId`
- SubLink Backend: stores state, executes charges, and verifies entitlement

## 3. Chain / Token

- Chain: Base Sepolia
- RPC: `https://sepolia.base.org`
- Explorer: `https://sepolia-explorer.base.org`
- Token: USDC `0x036CbD53842c5426634e7929541eC2318f3dCF7e`

## 4. Identity Model

### 4.1 Dedicated Unlink account

Each subscription gets a plan-specific dedicated Unlink account.

- `derivationIndex = Number(BigInt(keccak256(toHex(planId))) % 1_000_000n)`
- `seed = keccak256(walletClient.signMessage("sublink:{planId}"))`
- `unlinkAccount = unlinkAccount.fromSeed({ seed, accountIndex: derivationIndex })`

### 4.2 Auth key

Subscribers derive one deterministic auth key from their EVM wallet:

- sign domain: `sublink-auth-v1`
- hash the signature with `keccak256`
- use the hash as a secp256k1 private key

`authKeyId` is the lowercase EVM address of this auth key.

The auth key is used only for proofs and bearer tokens, never to move Unlink funds.

## 5. Subscription Model

SubLink separates lifecycle state from access entitlement.

### 5.1 Entitlement

Access is controlled by:

- `paidThroughAt`

A subscriber currently has access if and only if:

- bearer token is valid
- token signer matches `subscription.authKeyId`
- subscription belongs to the requested `planId`
- `paidThroughAt` exists
- `paidThroughAt > now`

### 5.2 Statuses

- `pending_activation`
  - initial charge has not succeeded yet
  - no entitlement exists
  - chargeable
- `active`
  - renewable subscription with current/future entitlement
  - chargeable
- `past_due`
  - renewal failed and entitlement was not extended
  - chargeable
- `cancelled`
  - subscriber stopped future renewals
  - existing access can remain valid until `paidThroughAt`
- `cancelled_by_failure`
  - subscription stopped after max consecutive failures
- `completed`
  - spending cap reached
  - existing access can remain valid until `paidThroughAt`

### 5.3 Chargeable statuses

Billing cron and manual activation retry may charge:

- `pending_activation`
- `active`
- `past_due`

## 6. Core Flows

### 6.1 Creator setup

1. Creator registers with `POST /creators`
2. Receives `creatorId` and `apiKey`
3. Creates plans with `POST /plans`

### 6.2 Initial subscription

1. Subscriber derives dedicated Unlink account for `planId`
2. Subscriber derives auth key
3. Subscriber funds the dedicated account with USDC
4. Subscriber signs subscribe proof:
   - `sublink-subscribe-v1:{planId}:{unlinkAddress}:{authKeyId}`
5. Subscriber calls `POST /subscribe`
6. Backend verifies auth proof and uniqueness on `(planId, authKeyId)`
7. Backend creates a `pending_activation` subscription row
8. Backend immediately attempts the first charge

On success:

- status becomes `active`
- `chargeCount += 1`
- `lastChargedAt = chargedAt`
- `paidThroughAt = chargedAt + interval`
- `nextChargeAt = paidThroughAt`
- endpoint returns `201`

On failure:

- status stays `pending_activation`
- `paidThroughAt = null`
- `nextChargeAt = chargedAt + PENDING_ACTIVATION_RETRY_SECONDS`
- endpoint returns `402`
- response includes the existing `subscriptionId`

### 6.3 Retrying a failed activation

If a subscriber calls `POST /subscribe` again for the same `(planId, authKeyId)` while the existing row is:

- `status = pending_activation`
- `chargeCount = 0`
- `paidThroughAt = null`

then the backend must:

- reuse the existing subscription row
- verify the activation details still match that row
- immediately reattempt the first charge
- not create a duplicate subscription

If the retry succeeds, the endpoint returns `201` with the same `subscriptionId`.

### 6.4 Renewal billing

Cron selects rows where:

- `status IN ('pending_activation', 'active', 'past_due')`
- `next_charge_at <= now`
- `consecutive_failures < MAX_CONSECUTIVE_FAILURES`

On successful charge:

- `totalSpent += amount`
- `chargeCount += 1`
- `consecutiveFailures = 0`
- `lastChargedAt = chargedAt`
- `paidThroughAt = max(previousPaidThroughAt, chargedAt) + interval`
- `nextChargeAt = paidThroughAt`
- status becomes:
  - `completed` if spending cap reached
  - otherwise `active`

On failed charge:

- `totalSpent` unchanged
- `chargeCount` unchanged
- `paidThroughAt` unchanged
- `consecutiveFailures += 1`
- `nextChargeAt` becomes:
  - `chargedAt + PENDING_ACTIVATION_RETRY_SECONDS` if no entitlement has ever been created
  - otherwise `chargedAt + interval`
- status becomes:
  - `pending_activation` if no entitlement exists
  - `past_due` if entitlement existed
  - `cancelled_by_failure` at the failure threshold

### 6.5 Verification

1. Subscriber creates bearer token:
   - format: `{subscriptionId}.{expiry}.{signature}`
   - message: `sublink-bearer-v1:{subscriptionId}:{expiry}`
2. Creator site forwards that token to `GET /verify/:planId` with `X-Api-Key`
3. Backend verifies:
   - creator owns the plan
   - token is valid
   - recovered signer matches `subscription.authKeyId`
   - subscription belongs to the plan
   - `paidThroughAt > now`
4. If yes: return `200 { valid: true, subscriptionId, planId }`
5. Otherwise: return `402` discovery payload

### 6.6 Cancellation

Subscriber can cancel only chargeable subscriptions:

- `pending_activation`
- `active`
- `past_due`

Cancellation stops future renewals but does not revoke already-paid entitlement.

## 7. API

### Public / creator

| Method | Path | Auth | Notes |
| --- | --- | --- | --- |
| POST | `/creators` | none | Register creator |
| GET | `/creators/:id` | none | Creator public info |
| GET | `/creators/by-evm/:evmAddress` | none | Creator public info |
| POST | `/plans` | none | Create plan |
| GET | `/plans` | none | List active plans |
| GET | `/plans/:id` | none | Plan with creator info |

### Subscriber

| Method | Path | Auth | Notes |
| --- | --- | --- | --- |
| POST | `/subscribe` | subscribe proof | Creates or retries pending activation; returns `201` on successful first charge or `402` on failed activation |
| GET | `/subscriptions` | Bearer | List subscriptions owned by recovered `authKeyId` |
| DELETE | `/subscriptions/:id` | Bearer | Cancel a chargeable subscription if owned by caller |
| GET | `/charges/:subscriptionId` | Bearer | Return charge history if owned by caller |

### Creator verification

| Method | Path | Auth | Notes |
| --- | --- | --- | --- |
| GET | `/verify/:planId` | `Authorization: Bearer {token}` + `X-Api-Key` | Returns `200` only when entitlement is currently valid |

### Admin

| Method | Path | Auth | Notes |
| --- | --- | --- | --- |
| POST | `/admin/run-cron` | `X-Admin-Key` | Run cron once manually |

### 8.1 subscriptions

Key columns:

- `status`

- `charge_count`
- `consecutive_failures`
- `last_charged_at`
- `paid_through_at`
- `next_charge_at`

Constraints:

- `UNIQUE(plan_id, auth_key_id)`

## 9. Config

- `MAX_CONSECUTIVE_FAILURES`
  - default `3`
- `PENDING_ACTIVATION_RETRY_SECONDS`
  - default `300`
- `BEARER_TOKEN_MAX_AGE_SECONDS`
  - default `86400`
- `BEARER_TOKEN_CLOCK_SKEW_SECONDS`
  - default `30`

## 10. Security / Privacy Rules

- Creator sites never receive subscriber wallet addresses
- Creator sites never receive Unlink account keys
- Backend stores dedicated Unlink account keys so it can execute recurring charges
- Bearer tokens are stateless and recover `authKeyId` from signature
- Verification requires both creator proof (`X-Api-Key`) and subscriber proof (`Bearer`)

## 11. Verification Guarantee

Creator verification answers:

- "Does this caller currently have paid entitlement for this plan?"

It does not answer:

- who the subscriber is in public wallet terms
- whether the next renewal will succeed
- how much dedicated private balance remains
